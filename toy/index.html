What this is / isn't — read before you trust a number

This is an analytic design-search toy, not a measurement. It ports the steady-state per-server model from examples/scale_model.rs verbatim. The numbers are "can the design close?" outputs, not telemetry from a running federation.

  • Steady-state, single-pool. One held-bytes pool per node scored by trust × demand × recency; no separate "archive vs cache" tiers, no warm-up transient. The eviction sweeper holds disk at 92% full.
  • Fountain replication. A holder stores one RaptorQ symbol = 1/N = 1/20 of a content; the swarm converges to H=30 holders; network overhead is H/N = 1.5× (not whole-copy 5×). Replication count is decoupled from trust radius.
  • The locality dividend is structural. self/family cohort content never emits a holds_bytes attestation, so no peer can request it — it's free inter-host. Only publishable() bytes (community + affiliations + species + planet + federation) cost anything to federate.
  • Empirical constants (sign/verify µs, scrub/canon rates, 14 KiB H3ERE trace) are baked in from CIRISVerify / CIRISEdge / CIRISPersist benchmarks — see FSD §2.
  • The cohort + depth knobs override the scenario. Changing them substitutes a different trust topology into the scenario's workload to let you feel sensitivity — they aren't the scenario's shipped defaults unless you pick "scenario default".
  • The gate sliders let you ask "what if the commodity box were beefier?" — the v1 bet is specifically the 1 TiB / 1 Gbps / 1 core column.

If a number looks wrong, the model is the source of truth — dispute the inputs, they're all visible.
